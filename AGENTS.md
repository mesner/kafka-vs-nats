# kafka-vs-nats

Benchmarks Kafka and NATS/JetStream across throughput, batch drain time, and throughput
saturation. Uses realistic HL7 v2.6 medical-device message payloads generated by `mock_message.py`.

## Running

```
uv run runner.py
```

Kafka and NATS-server are started and stopped automatically. Results print to the console and
are saved to `results/run_YYYYMMDD_HHMMSS.json`.

## Files

| File | Purpose |
|---|---|
| `mock_message.py` | HL7 message generators — do not modify |
| `config.py` | All tuneable constants (counts, rates, producer configs) |
| `metrics.py` | `MetricsCollector`, `ScenarioResult`, `stamp()` helper |
| `kafka_bench.py` | confluent-kafka benchmarks (synchronous) |
| `nats_bench.py` | nats-py benchmarks (asyncio) |
| `runner.py` | Orchestrator, table printer, JSON output |
| `services.py` | Starts/stops Kafka and NATS-server via subprocess |

## Scenarios

- **DEC burst** — 5,000 × ~4 KB HL7 numeric-vitals messages, sent as fast as possible.
  Tested with both `safe` (`acks=all`, `linger=0`) and `fast` (`acks=1`, `linger=5ms`, lz4)
  Kafka producer configs.
- **WCM burst** — 500 × ~175 KB HL7 waveform messages (12-lead ECG), sent as fast as possible.
- **Batch drain** — 5,000 DEC messages pre-loaded; consumer drain time measured from a cold start.
  Most representative of the target use case.
- **Saturation ramp** — Producer rate stepped through `SATURATION_RATES` (default: 50–800 msg/s).
  Stops early if consumer lag exceeds `SATURATION_LAG_LIMIT`. Extend rates in `config.py` to
  find the throughput ceiling.

## Key tuning knobs (`config.py`)

```python
DEC_BURST_COUNT    = 5_000
WCM_BURST_COUNT    = 500
BATCH_DRAIN_COUNT  = 5_000
SATURATION_RATES   = [50, 100, 200, 400, 800]   # msg/s — extend to find ceiling
SATURATION_STEP_DURATION = 15                    # seconds per step
```

## Caveats

- **Kafka burst latency is not representative.** The synchronous consumer poll loop trails the
  producer thread in burst scenarios, causing per-message latencies in the seconds range. This is a
  benchmark architecture artifact, not a Kafka limitation. Batch drain throughput is a better proxy
  for real-world performance.
- **Saturation ceiling not yet found.** Both systems handled 800 msg/s with zero lag. Raise
  `SATURATION_RATES` (try `[800, 2000, 5000, 10000]`) to find where each system starts falling
  behind.
- **Single-partition Kafka.** All tests use one partition (`replication_factor=1`). Multi-partition
  tests would show higher Kafka throughput at the cost of ordering guarantees.
- **NATS file storage.** Streams use `StorageType.FILE` for parity with Kafka's durable log.
  Switch to `StorageType.MEMORY` for a speed-of-light NATS upper bound.
